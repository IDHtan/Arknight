# Arknight 开发文档

这份文档是给策划、美术、技术美术使用的。

目标不是解释 C++ 细节，而是帮助你们知道：

- 当前项目在 Unreal Editor 里应该怎么继续开发
- 哪些工作可以直接在蓝图、DataTable、UMG 里完成
- 哪些地方必须与程序配合
- 如何提前把动画、图片、网格体、UI 资源准备好，等程序后续接逻辑

当前项目还没有正式的敌人系统，因此这份文档先围绕以下内容展开：

- 干员
- 子弹
- 地块
- 地图表
- UMG UI
- 调试地图
- 未完成动画与未来资源准备

---

## 1. 先理解 Unreal 的通用开发模式

当前项目采用的是 Unreal 很常见的一种模式：

`C++ 基类负责逻辑，Blueprint 子类负责资源、参数和表现。`

你可以把它理解成：

- 程序先写一个“能运行的空模板”
- 策划/美术再基于这个模板创建蓝图子类
- 蓝图子类里补模型、图片、数值、动画、UI 排版
- 最后把这些蓝图资产放进 DataTable、Widget、地图或父蓝图里使用

这也是 Unreal 官方长期推荐的组合方式之一：基础逻辑在 C++，可迭代的数据和表现更多交给 Blueprint 和数据资产处理。

### 1.1 你们平时主要做什么

对策划/美术来说，后续工作大多会落在以下位置：

- `Blueprint Class Defaults`：改参数、改资源引用
- `Components`：加模型、改缩放、改材质、改位置
- `Widget Blueprint Designer`：排版、换图、调动画
- `DataTable`：配置干员字典、地图布局
- `关卡/地图`：放测试对象、切调试关卡入口

### 1.2 一条标准工作流

以后你们做新内容，尽量按下面顺序：

1. 找到对应的 C++ 基类
2. 创建它的 Blueprint 子类
3. 在 Blueprint 里补资源和参数
4. 把它登记到 DataTable 或父 Widget 中
5. 进调试图验证

### 1.3 什么适合你们改，什么不适合

适合策划/美术直接改：

- 静态网格体、材质、图标、贴图
- 文字、描述、职业、费用、基础数值
- 攻击范围数组
- 地块外观和逻辑类型
- DataTable 内容
- UMG 排版、图片、按钮外观、Widget 动画
- 蓝图事件里的纯表现逻辑

不要自己改，或者改前先问程序：

- `Source/Arknight` 下的 C++ 文件
- 核心类继承关系
- 计时器、部署流程、索敌逻辑、伤害逻辑
- SaveGame 结构
- UMG 的控件变量名
- 未来 Spine2D 插件接入方案

### 1.4 当前项目的目录分工

推荐你把资源继续放在这些地方：

- 干员：`Content/Operators/<干员名>/`
- 地块：`Content/BattleMap/Cells/`
- UI：`Content/BattleMap/UIs/`
- 数据表：`Content/Datas/`
- 地图：`Content/Maps/`

现有模板资产可以优先参考：

- 干员模板：`Content/Operators/TEST/BP_TEST_Operator.uasset`
- 远程干员模板：`Content/Operators/TEST/BP_TEST_OperatorFar.uasset`
- 子弹模板：`Content/Operators/TEST/BP_TEST_Bullet.uasset`
- 地块模板：`Content/BattleMap/Cells/BP_TEST_*.uasset`
- HUD 模板：`Content/BattleMap/UIs/WBP_BattleHUDWidget.uasset`
- 调试入口：`Content/BattleMap/BattleDebug/BP_TEST_GM.uasset`

## 2. 最重要的通用规则：先有 C++ 基类，再做 Blueprint 子类

创建新内容时，不要直接随便新建一个空蓝图 Actor 再自己拼逻辑。

正确做法是：

1. 在 Content Browser 里找到对应 C++ 类
2. 右键
3. 选择 `Create Blueprint Class Based On ...`
4. 用这个 Blueprint 子类做后续编辑

这样做的好处：

- 逻辑不会丢
- 程序暴露出来的参数都能直接改
- 后续代码更新时，蓝图子类还能继续继承新逻辑

例如：

- 干员要从 `AOperatorBase` 派生
- 子弹要从 `ABulletBase` 派生
- 可部署地块要从 `ADeployableCell` 派生
- 资源地块要从 `AResourceCell` 派生
- 普通地块要从 `ABattleCell` 派生
- UI 要从对应的 `UUserWidget` C++ 类派生

## 3. UMG 是最特殊的一块：不仅要建子类，还要保证控件名字完全一致

这一条非常重要。

当前项目的 UI 不是“随便拖几个控件出来”就能工作，而是 C++ 通过 `BindWidget` 和 `BindWidgetAnim` 去查找 Widget Blueprint 里的控件和动画。

这意味着：

- Widget Blueprint 必须基于正确的 C++ 类创建
- 里面的控件名字必须和 C++ 里的变量名完全一致
- 动画轨道名字也必须和 C++ 里的动画变量名完全一致

如果名字不一致，常见后果是：

- 指针为空
- 运行时面板不显示
- 按钮没反应
- 动画不播放
- 甚至直接崩溃

### 3.1 先理解 Widget Blueprint 编辑器

Unreal 官方对 Widget Blueprint 的工作流可以概括成三块：

- `Designer`：排版和视觉层
- `Graph`：蓝图逻辑层
- `Animations`：UI 动画轨道

对你们来说，主要会用到：

- `Designer` 改布局和图片
- `Animations` 做滑入、淡入、缩放等 UI 动画
- `Graph` 只在需要少量表现逻辑时使用

当前项目更推荐的原则是：

- 游戏规则逻辑尽量交给 C++
- UI 的表现和排版交给 Widget Blueprint

### 3.2 `BindWidget` 规则

例如 C++ 写了：

```cpp
UPROPERTY(meta = (BindWidget))
UButton* ExitButton;
```

那么在对应的 Widget Blueprint 里，必须真的有一个名为 `ExitButton` 的按钮控件。

不是“意思差不多”就行，必须完全同名。

例如下面这些都不行：

- `BtnExit`
- `Exit_Btn`
- `ButtonExit`

只能是：

- `ExitButton`

### 3.3 `BindWidgetAnim` 规则

例如 C++ 写了：

```cpp
UPROPERTY(Transient, meta = (BindWidgetAnim))
UWidgetAnimation* SlideInAnim;
```

那么在 Widget Blueprint 里，动画轨道名字也必须是：

- `SlideInAnim`

如果你把轨道命名成：

- `SlideIn`
- `PanelIn`
- `ShowAnim`

那 C++ 就找不到。

### 3.4 UMG 的实际工作建议

当前项目里，策划/美术在 UMG 上最适合做的是：

- 调整面板布局
- 加入排版控件
- 导入并替换图片
- 设置字体、颜色、边距、尺寸
- 创建/修改 UMG 动画
- 调整按钮外观和进度条风格

不建议你们自己加太多 Graph 里的游戏规则逻辑，因为：

- 现在项目的 UI 逻辑已经和 C++ 绑定得比较深
- 图省事临时写在 Widget Graph 里的规则，后续最容易和代码打架

## 5. 干员开发流程

当前项目最重要的可扩展内容之一就是干员。

### 5.1 干员基类

干员的 C++ 基类是：

- `AOperatorBase`

推荐做法：

- 不要从空 Actor 开始做
- 直接创建 `AOperatorBase` 的 Blueprint 子类
- 或复制现有的 `BP_TEST_Operator` / `BP_TEST_OperatorFar` 作为模板再修改

### 5.2 干员蓝图里已有的核心组件

`AOperatorBase` 默认已经有：

- `SpriteComponent`
- `TargetingComp`

它们的作用分别是：

- `SpriteComponent`
  - 当前项目里实际是 `UStaticMeshComponent`
  - 名字虽然叫 Sprite，但现在是拿来挂 3D 网格体的
  - 这里负责显示角色本体

- `TargetingComp`
  - 负责攻击范围、攻击力、攻击速度、伤害类型等战斗参数
  - 这个组件上的数值直接影响战斗

### 5.3 干员蓝图里最常改的内容

在干员蓝图默认值里，你们通常会改这些参数：

| 参数 | 作用 |
|---|---|
| `OperatorName` | 干员唯一名字 |
| `OperatorDescription` | 描述文本 |
| `AvatarImage` | 卡面、详情面板、部署面板都会用 |
| `DeploymentCost` | 部署费用 |
| `DeployType` | 近战 / 远程 / 任意 |
| `RedeployTime` | 撤退后再部署冷却时间 |
| `AttackType` | 近战 / 远程 |
| `OperatorClassType` | 职业类型 |
| `VisualPlacementOffsetZ` | 自动贴地后的微调高度 |

在 `TargetingComp` 组件上，你们通常会改这些参数：

| 参数 | 作用 |
|---|---|
| `BaseAttackRange` | 相对攻击范围 |
| `MaxTargets` | 同时攻击目标数 |
| `AttackPower` | 攻击力 |
| `AttackSpeed` | 攻击间隔 |
| `DamageType` | 物理 / 法伤 / 真伤 |

### 5.4 组件编辑建议

在 `SpriteComponent` 上，你们通常会做：

- 设置 Static Mesh
- 调整位置
- 调整旋转
- 调整缩放
- 更换材质

注意：

- 当前部署贴地高度是自动计算的
- 如果角色看起来略微悬空或埋地，优先调 `VisualPlacementOffsetZ`
- 不要为了贴地效果去硬改很多奇怪的相对位置，先试这个参数

### 5.5 攻击范围怎么配

当前攻击范围来自 `TargetingComp->BaseAttackRange`，它是一个坐标数组。

例如：

- `(1, 0)` 表示前方 1 格
- `(2, 0)` 表示前方 2 格
- `(1, 1)` 表示右前方 1 格

它的本质是“相对坐标模板”，部署后会根据朝向变成实际范围。

因此：

- 策划可以直接在蓝图里配这个数组
- 不需要改 C++

### 5.6 近战干员与远程干员

如果是近战干员：

- `AttackType = Melee`
- 一般不需要 `BulletClass`

如果是远程干员：

- `AttackType = Ranged`
- 必须指定 `BulletClass`

### 5.7 当前项目里没有 `bulletStartPoint`

这一点和你给的 Enemy 示例不同，当前项目里还没有专门的子弹发射点组件。

现在远程子弹的生成位置是：

- 干员 Actor 的世界位置，也就是 `GetActorLocation()`

这意味着：

- 当前还不能在蓝图里精确摆“枪口点”
- 如果你觉得子弹从身体中心飞出来不好看，这属于后续程序要补的功能

你们现在能做的是：

- 先把子弹本身做出来
- 先把干员本体表现做好
- 如果以后要支持枪口点，再让程序补 `SceneComponent` 或 Socket 发射点

### 5.8 新建干员后，还必须登记到 DataTable

这是最容易漏的一步。

只做出蓝图还不够，必须把它登记进：

- `Content/Datas/DT_OperatorDictionary`

否则系统不会把它加入全局 roster。

## 6. 子弹开发流程

远程干员用到的子弹基类是：

- `ABulletBase`

### 6.1 创建方法

创建 `ABulletBase` 的 Blueprint 子类，或者参考：

- `Content/Operators/TEST/BP_TEST_Bullet.uasset`

### 6.2 可编辑内容

子弹蓝图里最常改的是：

| 参数 | 作用 |
|---|---|
| `MeshComp` | 子弹外观 |
| `FlySpeed` | 飞行速度 |

### 6.3 当前子弹系统的特点

目前的子弹系统比较轻量：

- 直线飞行
- 没有专门碰撞盒
- 没有轨迹修正
- 接近目标点时直接结算伤害

因此美术制作建议：

- 子弹模型不要太大
- 尽量用清晰的朝向
- 子弹轴向和模型前进方向尽量一致

如果以后子弹要做拖尾、命中特效、爆炸，那通常还需要程序后续补：

- Niagara
- 命中事件
- Socket 发射点
- 飞行曲线或抛物线

## 7. 地块开发流程

当前地块主要分三种：

- 普通地块：`ABattleCell`
- 可部署地块：`ADeployableCell`
- 资源地块：`AResourceCell`

### 7.1 普通地块

如果只是做一个“纯展示或阻挡”的格子，通常从：

- `ABattleCell`

派生即可。

要设置的重点：

| 参数 | 作用 |
|---|---|
| `CellMeshComponent` | 格子显示模型 |
| `LogicalCellType` | 逻辑类型 |

`LogicalCellType` 当前可用枚举包括：

- `DeployableGround`
- `BlockedGround`
- `DeployableHighGround`
- `BlockedHighGround`
- `ResourceSpawner`
- `Functional`

### 7.2 可部署地块

可部署地块应该从：

- `ADeployableCell`

派生。

核心要点：

- 是否允许部署，不是看美术外观，而是看 `LogicalCellType`
- 近战一般应设置为 `DeployableGround`
- 远程一般应设置为 `DeployableHighGround`

当前 `CanDeploy()` 的判断规则是：

- 近战干员只能上 `DeployableGround`
- 远程干员只能上 `DeployableHighGround`
- `Any` 可以无条件通过

### 7.3 可部署地块的蓝图表现

`ADeployableCell` 里有两个专门给蓝图做表现的事件：

- `EnableVisualEffect()`
- `DisableVisualEffect()`

它们是非常适合技术美术/美术接表现的地方。

你们可以在蓝图里做：

- 发光
- 地面描边
- 材质切换
- 粒子出现/消失
- 可部署时高亮，不可部署时恢复

这部分不需要改 C++。

### 7.4 资源地块

资源地块应该从：

- `AResourceCell`

派生。

它除了继承地块能力，还带有生命值和资源掉落逻辑。

常改参数：

| 参数 | 作用 |
|---|---|
| `ResourceType` | 木头 / 石头 / 金属 / 粮食 |
| `DropAmount` | 总掉落次数 |
| `DegradeToCellClass` | 被采空后替换成什么地块 |

同时还需要选中 `HealthComp` 组件，设置：

| 参数 | 作用 |
|---|---|
| `MaxHealth` | 最大生命 |
| `Defense` | 防御 |
| `ArtsResistance` | 法抗 |

### 7.5 资源地块的使用建议

例如你想做“木材资源点”：

1. 创建 `AResourceCell` 子蓝图
2. 设置 `ResourceType = Wood`
3. 设置 `DropAmount`
4. 设置 `HealthComp` 数值
5. 设置 `DegradeToCellClass`

如果你希望它被采空后变成可部署地面，就让：

- `DegradeToCellClass = 某个可部署地块蓝图`

- `Content/Datas/DT_OperatorDictionary`
- `Content/Datas/DT_BattleMaps`

它们都非常关键。

## 9. `DT_OperatorDictionary`：干员字典表

这个表的结构来自：

- `FOperatorDictionary`

字段有：

| 字段 | 作用 |
|---|---|
| `OperatorName` | 干员名字 |
| `OperatorClass` | 干员蓝图类 |

### 9.1 最重要的规则：行名要与 `OperatorName` 一致

当前代码里有一个很关键的实现细节：

- 系统后续会用 `OperatorName` 再去查 DataTable 行

因此请务必遵守：

- `Row Name == OperatorName`

例如你做一个干员 `TEST`，正确做法是：

- Row Name：`TEST`
- `OperatorName`：`TEST`
- `OperatorClass`：`BP_TEST_Operator` 或其他干员蓝图

不要这样写：

- Row Name：`Operator_001`
- `OperatorName`：`TEST`

否则后续 roster 组装可能失败。

### 9.2 新增一个干员的完整步骤

1. 先做干员蓝图
2. 打开 `DT_OperatorDictionary`
3. 新增一行
4. 行名填写干员名
5. `OperatorName` 填同样的名字
6. `OperatorClass` 指向对应干员蓝图
7. 保存

### 9.3 新干员何时会被系统识别

当前 `URougeliteRunSubsystem::CheckOperators()` 会在初始化时读取这个表。

也就是说：

- 新干员加进表后，重新进游戏或重启 PIE，系统就会尝试把它注册进存档等级表

## 10. `DT_BattleMaps`：战斗地图布局表

这个表的结构来自：

- `FLevelLayoutConfig`

字段有：

| 字段 | 作用 |
|---|---|
| `GridX` | 地图宽度 |
| `GridY` | 地图高度 |
| `LayoutData` | 每一格要生成什么地块类 |

### 10.1 最重要的规则：Row Name 就是关卡 ID

当前地图生成逻辑会直接用 `LevelID` 去找这一行。

因此：

- `DT_BattleMaps` 的 Row Name，就是地图逻辑 ID

例如：

- Row Name：`TEST_MAP_01`

那么调试关卡的 `LevelID` 也要写：

- `TEST_MAP_01`

### 10.2 `LayoutData` 的填写方式

当前代码按下面方式生成地图：

- 先 X，再 Y
- 索引 `i = 0` 时是 `(0, 0)`
- 同一行先从 `X = 0` 填到 `X = GridX - 1`
- 然后再换到下一行 `Y + 1`

你可以把它理解成：

- `LayoutData` 是一个按“行优先”的一维数组

所以请务必保证：

- `LayoutData` 的元素数量 = `GridX * GridY`

例如：

- `GridX = 3`
- `GridY = 2`

那么 `LayoutData` 必须有 6 个元素，顺序是：

1. `(0,0)`
2. `(1,0)`
3. `(2,0)`
4. `(0,1)`
5. `(1,1)`
6. `(2,1)`

### 10.3 `LayoutData` 里填什么

这里填的不是实例，不是名字，而是：

- `ABattleCell` 子类蓝图类引用

也就是类似：

- 某个 `BP_DeployableGround`
- 某个 `BP_BlockedGround`
- 某个 `BP_ResourceWood`

### 10.4 推荐的地图制作步骤

1. 先把所有要用的地块蓝图做出来
2. 打开 `DT_BattleMaps`
3. 新增一行，行名设成关卡 ID
4. 填 `GridX`
5. 填 `GridY`
6. 按顺序填满 `LayoutData`
7. 保存
8. 用调试地图进入战斗测试

## 11. 调试地图与测试流程

当前快速测试入口是：

- `Lvl_DebugMap`
- `BP_TEST_GM`

调试用的 C++ 基类是：

- `ALvlDebugGameMode`

它有一个参数：

- `LevelID`

### 11.1 调试时怎么用

1. 打开 `Lvl_DebugMap`
2. 确认当前 GameMode 使用的是对应调试蓝图
3. 在调试 GameMode 的 `LevelID` 里填你要测试的地图行名
4. 运行

运行时它会：

1. 把 `LevelID` 写进 `PendingBattleLevelID`
2. 切到 `Lvl_BattleMap`
3. 由战斗逻辑读取 `DT_BattleMaps` 生成地图

### 11.2 调试时最常检查什么

做完新内容后，至少检查下面这些：

干员检查：

- 卡片是否出现
- 头像是否显示
- 费用是否正确
- 是否能正常部署
- 攻击范围是否正确
- 远程是否能发子弹
- 撤退后是否能回到卡组

地块检查：

- 外观是否正确
- 逻辑类型是否正确
- 是否能被部署
- 资源地块是否能被打
- 采空后是否正常退化

UI 检查：

- 控件是否显示
- 动画是否播放
- 点击是否有反应
- 是否出现空白、错位、隐藏失败

## 13. UMG 的实际制作建议

这部分是给 UI 美术最重要的操作说明。

### 13.1 先做什么

如果对应的 Widget Blueprint 已经存在，优先做这些：

- 导入图片
- 替换 Brush
- 调整布局
- 调整字号与留白
- 调整锚点、对齐、Padding
- 创建或微调 UMG 动画

### 13.2 少做什么

除非程序说明可以，否则尽量不要自己做这些：

- 改控件变量名
- 改动画变量名
- 大量重写 Graph 逻辑
- 自己新增会影响主流程的按钮逻辑

### 13.3 关于子 Widget

像下面这些控件，不只是普通容器，而是“带 C++ 逻辑的子 Widget”：

- `ExitConfirmPanel`
- `OperatorDetailPanel`
- `CellDetailPanel`
- `DeployPanel`

因此做 HUD 时，不是随便拖一个普通 Panel 顶上去，而是要放对应的子 Widget 蓝图。

### 13.4 关于攻击范围小格子

`WBP_OperatorDetailWidget` 里的攻击范围可视化依赖：

- `GridBlockClass`

它本质上是一个“范围格子预制 Widget”。

所以如果你发现攻击范围不显示，优先检查：

- 这个类有没有指定
- 这个小格子 Widget 是否存在

### 13.5 关于 UMG 动画

当前项目已经在 C++ 里播放了两处 UMG 动画：

- `WBP_OperatorDetailWidget` 的 `SlideInAnim`
- `WBP_CellDetailWidget` 的 `SlideInAnim`

因此美术可以直接在对应 Widget Blueprint 里：

1. 打开底部 `Animations`
2. 新建动画轨道
3. 名字必须叫 `SlideInAnim`
4. 在 `Timeline` 里打关键帧
5. 调位置、透明度、缩放等属性

建议做法：

- 详情面板可以做轻量滑入 + 淡入
- 不要做过长动画，避免影响操作感
- 先保证信息可读，再做额外花哨效果

## 14. 动画与表现：当前哪些能直接做，哪些还不能

这一节很重要，因为很多表现工作可以先准备，但还不能完全在当前代码中自动调用。

### 14.1 当前已经能直接做的动画

当前已经可以直接做并立刻生效的，是：

- UMG 动画

也就是：

- 面板滑入

因为这部分已经有 `BindWidgetAnim` 和 `PlayAnimation()` 逻辑。

### 14.2 当前还没有代码调用入口的动画

下面这些世界内动画，目前还是“占位点”：

- 干员部署动画
- 干员攻击动画
- 干员撤退动画

原因是 `AOperatorBase` 里虽然已经留了注释位置，但还没真正接动画系统：

- `OnDeployed()` 里有 `Deploy Animation here`
- `OnRetreat()` 里有 `Retreat Animation here`
- `ExecuteAttack()` 里有 `Attack Animation here`

### 14.3 这不代表你们现在什么都不能做

你们现在完全可以先把素材准备好，只是“先准备资源”，等程序后续来接调用。

推荐做法如下。

## 15. 如何为未来动画逻辑提前准备资源

### 15.1 如果继续走当前 3D/网格体路线

当前 `AOperatorBase` 使用的是：

- `SpriteComponent`，实际类型是 `UStaticMeshComponent`

这意味着当前项目还没有正式的：

- SkeletalMesh 角色动画管线
- Animation Blueprint
- Montage 调用点

所以如果你们现在要为未来准备资源，建议：

1. 先把角色静态外观、图标、材质做完整
2. 把未来要用的动作资源单独整理在对应干员文件夹
3. 保持统一命名

推荐命名：

- `AN_Deploy`
- `AN_Idle`
- `AN_Attack`
- `AN_Retreat`

或者至少保证 DCC 原文件、导出文件、说明文档里沿用统一命名。

同时建议保持：

- 角色脚底为统一参考原点
- 模型正面朝向统一
- 攻击动作记录预期命中时刻

这样程序以后在接入动画播放时更轻松。

### 15.2 如果未来切到 Spine2D

这一点要特别注意：

- 当前项目还没有现成的 Spine2D 运行时接入代码
- 如果要正式上 Spine2D，通常需要安装对应插件
- 插件版本必须和当前 Unreal 版本匹配

所以在真正开做 Spine2D 量产前，建议先确认三件事：

1. 具体使用哪个 Spine 运行时插件
2. 插件是否支持当前 UE 版本
3. 最终是用骨骼动画组件、Paper2D、还是自定义渲染组件接入

在插件和技术方案没定之前，美术可以先做这些准备：

- 统一动作状态命名
- 统一导出规范
- 统一角色朝向
- 统一锚点/脚底基准

推荐状态命名：

- `Idle`
- `Deploy`
- `Attack`
- `Retreat`
- `Hit`

### 15.3 UMG 动画怎么提前做

UMG 的准备最简单，因为它已经能直接被代码调用。

建议你们在 Widget Blueprint 里提前做好：

- `SlideInAnim`
- 面板开关时的透明度变化
- 详情面板的轻量位移动画

后续程序如果要补更多调用，也只需要：

- 再加新的 `BindWidgetAnim`
- 再在对应函数里 `PlayAnimation()`

### 15.4 当前最适合技术美术先做的表现点

现在最适合提前打磨的，是：

- `ADeployableCell` 的可部署高亮效果
- UMG 的滑入动画
- 子弹模型与材质
- 干员头像、职业图标、UI 图标
- 资源地块的外观区分

## 18. 当前未完成、但你们现在就该知道的地方

这部分不是让你们去修 C++，而是让你们知道哪些系统还在半成品状态。

### 18.1 战斗结束结算

`URougeliteRunSubsystem::ConcludeBattle()` 目前还是空的。

这意味着：

- 退出战斗后的完整结算流还没真正完成

对策划影响：

- 暂时不要把正式结算流程当成已完成系统来设计长链路

### 18.2 部署、攻击、撤退动画

`AOperatorBase` 里这三处世界内动画还没接：

- 部署
- 攻击
- 撤退

对美术影响：

- 可以先准备资源
- 但当前别指望进游戏就自动播

### 18.3 子弹发射点

当前没有 `bulletStartPoint` 这类组件。

对美术影响：

- 远程子弹暂时从角色位置发出
- 如果以后要更自然的枪口/法杖出弹，需要程序补发射点系统

### 18.4 可部署格高亮

这块已经有蓝图事件入口，但表现体系还没完全成型。

对技术美术影响：

- 现在就可以在蓝图里开始做高亮效果
- 后续如果程序补更多状态，也比较容易继续接
